# Spring Transaction 이해


## 1. DB 세션

트랜잭션을 이해하기 앞서 DB 세션이라는 것을 알아보자.

> **세션**
> 데이터베이스에 커넥션이 생성됐을 때 데이터베이스 내부에 생성되는 것. 가령 커넥션풀에 커넥션이 10개 있다면 세션도 10개가 생성된다. 

세션의 역할은 클라이언트(애플리케이션, DB 관리 툴 등)를 통해 데이터베이스 서버에 요청한 것을 처리하는 것이다. 

세션이 요청을 처리하는 단위를 **트랜잭션(Transaction)** 이라 한다. 세션은 트랜잭션을 시작하고 커밋 또는 롤백으로 트랜잭션을 종료할 수 있다. 


## 2. 트랜잭션의 일반적인 위치

트랜잭션은 비즈니스 로직이 있는 서비스 계층에서 걸어주는게 일반적이다. 비즈니스 로직이 잘못되었을 때 해당 비즈니스 로직으로 잘못된 부분을 함께 롤백해야되기 때문이다.  

트랜잭션을 시작하기 위해서는 커넥션이 필요하기 때문에 이것은 즉 서비스 계층에서 커넥션을 획득해야된다는 것을 의미한다. 그리고 커넥션은 세션을 만들게 되므로 **같은 세션을 사용하기 위해서는 같은 커넥션을 유지**해야된다.  

그런데 서비스 계층에 트랜잭션을 사용하기 위해서는 트랜잭션을 사용하기 위한 코드들을 넣어야 한다. 이 코드들은 꽤 복잡하기도 하고 결과적으로 서비스 계층의 코드가 매우 복잡해지는 결과를 초래한다. 서비스 계층의 코드는 비즈니스 로직만을 가지고 있는 것이 좋은 설계이다.  


## 3. 트랜잭션 추상화

서비스 계층에서 트랜잭션을 시작하기 위해서는 사용하고 있는 데이터접근 기술에서 사용하는 트랜잭션 방식을 따라야 한다. 이 말은 데이터접근 기술이 변경되면 그에 따른 트랜잭션을 시작하는 방식도 변경해야된다는 것을 의미한다.

<p align="center">출처: 인프런 김영한님 Spring DB 1편 강의자료 </p>


이것은 OOP의 SOLID 원칙 중 단일책임원칙에도 어긋난다. 데이터접근 기술을 변경한 것인데 연쇄적으로 다른 곳에서도 변경이 일어나기 때문이다.  

늘 그랬듯이 이런 경우에는 추상화를 하면 된다. 비즈니스 로직을 담고 있는 서비스 계층은 트랜잭션을 담당하는 인터페이스를 바라보게 하고 각각의 데이터접근 기술에 따라 인터페이스를 구현하는 구현체를 DI를 통해서 사용하면 되는 것이다.  

스프링을 사용하면 해당 인터페이스를 만들 필요가 없다. 이미 만들어 두었기 때문이다. 스프링에서 트랜잭션을 담당하는 인터페이스는 **PlatformTransactionManager** 이다. 그리고 이미 여러 데이터접근 기술에 따른 구현체도 만들어져 있다.

<p align="center"><img width="700" alt="추상화 후" src="https://user-images.githubusercontent.com/100478841/209597524-7f9c5a0e-74b5-4688-8654-199c099d41fa.png"></p>
<p align="center">출처: 인프런 김영한님 Spring DB 1편 강의자료</p>

**PlatformTransactionManager**는 `getTransaction()`, `commit()`, `rollback()` 세 가지 메서드를 가지고 있다. getTransaction()은 트랜잭션을 시작한다는 것이고, commit()과 rollback()은 이름 그대로 커밋과 롤백을 하는 역할을 수행한다. PlatformTransactionManager 인터페이스 및 구현체를 통상 **트랜잭션 매니저**라고 부른다.


## 4. 트랜잭션 동기화 

트랜잭션 동기화를 위해서는 동일한 데이터베이스 커넥션을 유지해야한다. 스프링은 이러한 작업을 위해 **트랜잭션 동기화 매니저**라는 것을 제공한다. 트랜잭션 동기화 매니저는 생성된 커넥션을 보관하고 있는 곳이라고 생각하면 된다. 

<p align="center"><img width="700" alt="동기화 매니저" src="https://user-images.githubusercontent.com/100478841/209602777-3483f1c8-2b5e-40eb-8fb7-dce40aabf935.png"></p>
<p align="center">출처: 인프런 김영한님 Spring DB 1편 강의자료</p>
**동작방식**
1. 트랜잭션 매니저(PlatformTransactionManager 및 구현체)가 데이터소스를 통해 커넥션을 만들고 트랜잭션을 시작한다. 
2. 트랜잭션 매니저가 트랜잭션이 시작된 커넥션을 트랜잭션 동기화 매니저에 보관한다. 
3. 레포지토리에서는 트랜잭션 동기화 매니저에 보관된 커넥션으로 DB에 접근한다. 트랜잭션 동기화 매니저에 있는 커넥션을 가져오려면 `DataSourceUtils.getConnection()` 을 해주면 된다.
4. 트랜잭션이 종료되면 트랜잭션 매니저가 트랜잭션 동기화 매니저에 보관된 커넥션을 통해 트랜잭션을 종료하고 커넥션을 닫는다. 


## 5. 트랜잭션 AOP

트랜잭션 AOP를 보기 전에 트랜잭션 AOP를 왜 알아야 하는지를 알아보면 다음과 같다.  

트랜잭션 추상화, 트랜잭션 동기화 등 트랜잭션을 보다 더 나은 방법으로 이용할 수 있게 되었지만 결정적으로 해결하지 못 한 문제가 하나 있다. 바로 **트랜잭션을 이용하기 위한 코드가 비즈니스 로직을 담당하는 서비스 계층에 계속해서 남아있다는 것**이다. 일반적으로 좋은 설계는 서비스 계층에는 비즈니스 로직만을 두는 것이다.  

트랜잭션 AOP는 트랜잭션을 처리하는 과정을 별도의 프록시에서 처리하고 서비스 계층에선 비즈니스 로직만을 처리하기 위해 사용하는 것이다.

그렇다면 트랜잭션 AOP를 적용하는 것은 어떻게 해야될까? 개발자가 해야될 건 많지 않다. 트랜잭션 처리가 필요한 곳에 **@Transactional** 어노테이션만 붙여주면 된다. **@Transactional** 어노테이션은 메서드에 붙여도 되고 클래스에 붙여도 된다. 클래스에 붙이는 경우에는 해당 클래스의 public 메서드 모두가 AOP 적용 대상이 된다.
