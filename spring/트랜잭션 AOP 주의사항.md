트랜잭션 AOP 주의사항 - 프록시 내부 호출
---

## 1.  트랜잭션 정상적으로 동작

Class에 @Transactional이 붙어있거나 클래스 내부의 메서드 중 하나라도 @Transactional이 붙어있으면 스프링의 트랜잭션 AOP가 적용되면서 **프록시 객체**를 만든다.

이렇게 함으로써 트랜잭션에 관한 처리는 프록시 객체에서 전부 담당하고 실제 객체(target)는 비즈니스 로직만 담당하는 역할의 분리를 달성할 수 있다. 

스프링 트랜잭션 AOP를 활용하여 프록시 객체를 만들면 스프링은 실제 객체 대신 프록시 객체를 스프링 빈으로 등록한다. 주입 받아서 사용할 때도 실제 객체를 주입 받는 것이 아니라 등록된 프록시 객체를 주입받게 된다. **프록시 객체는 실제 객체를 상속하여 만들어진 것이므로 다형성을 활용하여 실제 객체 타입으로 선언해줄 수 있다.**

정상적으로 트랜잭션이 적용되어서 코드를 호출하는 경우는 다음과 같이 진행된다.

> 1. 클라이언트가 특정 객체의 메서드를 호출한다. (이 때 이 객체는 프록시 객체를 가지고 있음을 가정한다.)	
> 2. 프록시 객체가 우선적으로 호출된다.
> 3. 메서드에 @Transactional이 있다면 트랜잭션이 정상적으로 적용된다.
> 4. 트랜잭션 적용이 완료되면 실제 객체(target)에 있는 메서드를 호출한다.  
> 5. 실제 객체에 있는 메서드가 처리를 완료하면 트랜잭션이 종료된다.


## 2. 정상적으로 동작하지 않는 경우

정상적으로 동작하지 않는 경우의 대표적인 경우는 프록시 내부에 @Transactional이 적용되지 않는 메소드 (가령 A) 내부에 @Transactional이 적용된 메소드 (가령 B)가 있고 A가 B를 호출할 때 문제가 발생하는 경우이다.

호출의 흐름을 표현하자면 다음과 같다.

> 1. 클라이언트가 특정 객체의 메소드를 호출한다. (이 객체는 프록시 객체를 가지고 있음을 가정한다.)
> 2. 프록시 객체가 우선적으로 호출된다.
> 3. 프록시 객체 내부에 A 메서드를 호출한다. (@Transactional 적용 X)
> 4. @Transactional이 없기 때문에 트랜잭션이 적용되지 않는다.
> 5. 트랜잭션이 적용되지 않은 채로 실제 객체(target)가 호출된다. 
> 6. 실제 객체(target)가 @Transactional이 있는 B 메소드를 호출한다. -> **문제 발생지점**

6번에서 문제가 발생한다. A 메서드는 트랜잭션이 적용되지 않은 상태로 처리를 진행하는데 이 과정에서 트랜잭션이 적용되어 있는 B 메서드를 호출한다고 해도 B 메서드에는 트랜잭션이 적용되지 않기 때문이다. 


## 3. 해결방법

해결하는 방법이 다양하게 있지만 대표적으로 해결하는 방법으로는 트랜잭션이 적용되는 메서드를 별도의 클래스를 만들어서 그 안에 집어넣어 분리하는 방식이다. 이렇게 별도의 클래스를 만들고 스프링 빈으로 등록해준 다음 이것을 주입받아서 사용하면 아예 클래스가 분리되어있기 때문에 트랜잭션이 별도로 적용되어 관리된다는 장점이 있다.
