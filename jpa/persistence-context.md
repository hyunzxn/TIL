# 영속성 컨텍스트

<br>

## 영속성 컨텍스트란?

### 의미

엔티티를 저장하는 영구 저장 환경

: 애플리케이션과 데이터베이스 사이에서 데이터의 조회, 저장, 수정, 삭제 등을 관리하는 역할을 담당한다.
<br>

엔티티 매니저를 하나 만들 때마다 하나의 영속성 컨텍스트가 만들어진다. 즉 영속성 컨텍스트에 접근하기 위해서는 엔티티 매니저가 필요하다. 

<br>

### 내부구조

1. 1차 캐시

엔티티 매니저를 통해 영속성 컨텍스트에 접근해서 엔티티를 영속 상태로 만들면 모든 엔티티는 영속성 컨텍스트 내부의 1차 캐시에 저장이 된다. 1차 캐시에는 각 엔티티별로 @Id, 엔티티, 스냅샷 이렇게 세 개의 정보가 저장된다. 

<br>

2. 쓰기 지연 SQL 저장소

JPA는 엔티티를 영속 상태로 만들자마자 바로 DB에 SQL 쿼리를 날리지 않는다. 실질적으로 DB에 쿼리를 날리는 시점은 하나의 트랜잭션이 커밋이 되는 시점이다. 이렇게 함으로써 성능 개선의 여지를 남겨둘 수 있고 버퍼링을 사용할 수 있다. 

<br>

## 엔티티의 생명 주기

엔티티와 영속성 컨텍스트의 관계에서 엔티티는 4가지의 생명주기를 가진다. 

- 비영속(new): 영속성 컨텍스트와 아무런 관계도 없는 상태 Ex) 객체를 막 생성만 했을 때
- 영속(managed): 영속성 컨텍스트에 의해 관리되는 상태 Ex) em.persist(member);
- 준영속(detached): 영속성 컨텍스트에 저장되었다가 분리된 상태 Ex) em.detach(member);
- 삭제(removed): 삭제된 상태 

<br>

## 영속성 컨텍스트의 이점

영속성 컨텍스트를 사용하면 어떤 이점이 있을까?

<br>

1. 1차 캐시 조회

앞서 엔티티를 영속 상태로 만들면 1차 캐시에 모두 저장이 된다고 했다. 이렇게 함으로써 얻는 이득은 동일한 엔티티를 조회할 때 DB를 거치지 않고 1차 캐시에서 바로 조회할 수 있으므로 성능적으로 유리하다는 것이다. 그러나 이런 이점은 아주 크진 않은데 왜냐하면 1차 캐시에서 조회는 하나의 트랜잭션 단위로 이뤄지기 때문이다. 매우 짧은 순간인 트랜잭션 단위에서의 조회이기 때문에 엄청 큰 유의미한 차이가 있진 않다. 애플리케이션 전체에서 가져다 쓰는 것은 2차 캐시가 담당한다.

<br>

만약 1차 캐시에 저장되어 있지 않은 데이터를 조회하면 일단 DB에서 데이터를 조회하고 그것을 1차 캐시에 저장한 다음에 1차 캐시에서 데이터를 조회한다. 

<br>

2. 영속 엔티티 동일성 보장

JPA의 기본적인 철학은 데이터를 자바 컬렉션에 저장하고 빼서 쓰듯이 사용하자는 것이다. 따라서 영속성 컨텍스트에 저장하고 동일한 엔티티를 조회하면 모두 동일한 엔티티임이 보장된다. 

Ex)

```java
Member a = em.find(Member.class, "member1");
Member b = em.find(Member.class, "member1");
System.out.println(a == b);
// 결과: true
```

<br>

이런 특성을 조금 어려운 말로 `1차 캐시로 반복 가능한 읽기(REPEATABLE READ) 등급의 트랜잭 션 격리 수준을 데이터베이스가 아닌 애플리케이션 차원에서 제공`(출처: 김영한- 자바 ORM 표준 JPA 프로그래밍 자료) 한다고 한다. 

<br>

3. 트랜잭션을 지원하는 쓰기 지연

이것은 영속성 컨텍스트 내부의 쓰기 지연 SQL 저장소와 관련된 특성이다. 앞서 설명한 바와 같이 SQL 쿼리를 영속 상태로 만들 때 바로 날리지 않고 트랜잭션 커밋을 할 때 한 번에 모아서 보낸다. 

<br>

4. 변경감지(Dirty Checking)

엔티티 수정과 관련하여 영속성 컨텍스트는 또 편리하게 수정을 할 수 있도록 도와준다. 엔티티를 영속상태로 만들 때 1차 캐시에 아이디와 엔티티 그리고 스냅샷이 저장된다고 앞서 살펴본 바 있다. 변경 감지는 이 때 스냅샷을 이용한다. 

<br>

```java
Member member = new Member("1L", "A")

em.persist(member);

member.setName("B");
```

<br>

멤버 객체를 하나 만들고 그것을 영속 상태로 만들어줬다. 그리고 나서 객체의 값을 변경해줬다. 그렇다면 이렇게 하고 나서 다시 `em.persist(member);`를 해줘야 할까? 그렇지 않다. JPA의 기본철학은 컬렉션처럼 데이터를 자유롭게 저장하고 이용하자는 것이기 때문에 컬렉션에 저장된 값을 변경하도록 다시 컬렉션에 저장하지 않듯이 영속성 컨텍스트(1차 캐시)에 저장된 엔티티의 값을 바꿀 때는 값만 바꿔줘도 알아서 변경감지가 되어 트랜잭션이 커밋될 때 데이터베이스에 반영이 된다. 

<br>

## 플러시

플러시란 영속성 컨텍스트의 내용을 데이터베이스에 반영하는 동기화를 의미한다. 플러시가 발생하는 경우는 다음과 같다. 플러시는 다음과 같은 프로세스로 이뤄진다. 

<br>

- 데이터베이스에 저장된 데이터 변경 감지
- 수정시키는 SQL 쿼리가 쓰기 지연 SQL 저장소에 저장된다.
- 쓰기 지연 SQL 저장소에 있는 쿼리가 데이터베이스에 전송됨으로써 데이터 업데이트가 발생한다. 

<br>

플러시를 발생시키는 방법은 세 가지가 있다.

- em.flush()
- 트랜잭션 커밋 시 자동으로 플러시 호출
- JPQL 쿼리 실행 시 자동으로 플러시 호출 

보통 두번째 경우가 제일 일반적이다.
